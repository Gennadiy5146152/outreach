# ТЗ: рабочий экран "Требуют решения"

## 1. Назначение

Раздел "Требуют решения" — главный рабочий экран после отправки аутрича.

Его задача — не просто показать входящие письма, а ответить пользователю на вопрос:

> "Какие диалоги мне сейчас нужно обработать и что с ними делать дальше?"

Раздел должен собирать все ситуации, где автоматическая цепочка не должна продолжаться без ручного решения.

## 2. Какие диалоги попадают в раздел

В раздел попадают диалоги, если есть хотя бы одно условие:

- получен живой ответ от лида;
- ответ классифицирован как `positive_reply`;
- ответ классифицирован как `negative_reply`;
- ответ классифицирован как `neutral_reply`;
- ответ классифицирован как `not_target`;
- ответ классифицирован как `auto_reply`;
- ответ классифицирован как `unknown`;
- ответ не удалось уверенно привязать к конкретной рассылке;
- ответ привязан слабым способом: только по email и теме;
- у контакта есть несколько активных/исторических цепочек и нужна ручная проверка;
- цепочка была автоматически остановлена после ответа;
- follow-up ожидает решения пользователя: продолжать, остановить или перенести.

Не должны попадать в раздел:

- прогревочные письма;
- технические события;
- исходящие письма без входящего ответа;
- тестовые письма, если они не требуют ручного решения;
- уже закрытые диалоги со статусом `closed`, `not_target`, `unsubscribed`, `bounced`, если по ним нет нового входящего.

## 3. Группировка

Экран должен показывать не отдельные письма, а диалоги.

Единица работы:

- лид;
- email лида;
- компания;
- конкретная outreach-цепочка или импорт;
- связанная история отправленных и входящих писем.

Если у одного email несколько рассылок, сервис должен:

- пытаться связать ответ по `Message-ID`, `In-Reply-To`, `References`;
- показывать ответ в конкретной цепочке только при точной технической связи;
- при слабой связи показывать предупреждение;
- при неоднозначной связи не раскладывать ответ автоматически по кампаниям;
- отправлять такой диалог в "Требуют решения" с пометкой "нужно выбрать цепочку".

## 4. Карточка диалога

Карточка в списке должна показывать:

- компанию;
- email;
- контактное лицо;
- сегмент;
- название импорта/кампании;
- отправляющий mailbox;
- дату последнего входящего;
- тему последнего письма;
- короткий очищенный текст ответа без истории переписки;
- статус диалога;
- классификацию ответа;
- качество привязки:
  - `точно: ответ в ветке письма`;
  - `примерно: email и тема совпали`;
  - `примерно: у email была одна цепочка`;
  - `не привязано к рассылке`;
  - `привязано без отметки метода`;
- следующее действие;
- количество отправленных писем;
- количество входящих ответов;
- есть ли будущие follow-up в очереди.

## 5. Детальная карточка

При открытии диалога пользователь должен видеть:

- всю цепочку писем в хронологическом порядке;
- отправленные письма;
- follow-up письма;
- входящие ответы;
- очищенный текст ответа;
- полный текст ответа по кнопке "Показать полностью";
- к какому исходящему письму привязан ответ;
- качество привязки ответа;
- события по диалогу:
  - отправлено;
  - получен ответ;
  - цепочка остановлена;
  - классификация изменена;
  - follow-up продолжен;
  - follow-up перенесен;
  - добавлено в стоп-лист;
- текущий статус;
- заметку пользователя;
- рекомендуемое следующее действие.

## 6. Основные действия пользователя

В карточке диалога должны быть быстрые действия:

- `Ответить вручную`;
- `Остановить цепочку`;
- `Продолжить follow-up`;
- `Вернуться позже`;
- `Отметить как интерес`;
- `Отметить как отказ`;
- `Отметить как не целевой`;
- `Отметить как автоответ`;
- `Добавить email в стоп-лист`;
- `Добавить домен в стоп-лист`;
- `Выбрать правильную цепочку`;
- `Закрыть без действия`.

Действия должны быть понятны по последствиям:

- "Остановить цепочку" отменяет будущие pending/retrying follow-up.
- "Продолжить follow-up" разрешает следующий шаг цепочки.
- "Вернуться позже" переносит следующий follow-up на выбранную дату.
- "Отметить как интерес" переводит диалог в статус `positive` и оставляет задачу ручного ответа.
- "Закрыть без действия" убирает диалог из review, но не удаляет историю.

## 7. Статусы диалога

Минимальный набор статусов:

- `needs_review` — требует решения;
- `waiting_reply_review` — получен ответ, нужно решить;
- `manual_reply_needed` — нужен ручной ответ;
- `positive` — есть интерес;
- `negative` — отказ;
- `not_target` — не целевой;
- `auto_reply` — автоответ;
- `later` — вернуться позже;
- `continued` — follow-up продолжен;
- `stopped` — цепочка остановлена;
- `closed` — диалог закрыт;
- `bounced` — недоставка;
- `unsubscribed` — отписка/просьба не писать.

## 8. Следующее действие

Для каждого диалога нужно хранить и показывать `next_action`.

Примеры:

- `reply_manually`;
- `approve_followup`;
- `stop_sequence`;
- `choose_correct_thread`;
- `classify_reply`;
- `schedule_later`;
- `add_to_suppression`;
- `export_for_ai`;
- `no_action`.

Текст в интерфейсе должен быть по-русски:

- "Ответить вручную";
- "Решить, продолжать ли follow-up";
- "Выбрать правильную цепочку";
- "Классифицировать ответ";
- "Вернуться позже";
- "Добавить в стоп-лист";
- "Действий не требуется".

## 9. Правила остановки и продолжения цепочки

Когда приходит живой ответ:

- все будущие follow-up по этому диалогу переводятся в `paused_reply_received` или аналогичный статус;
- цепочка не продолжается автоматически;
- диалог попадает в "Требуют решения";
- пользователь должен явно выбрать действие.

Если пользователь нажал "Продолжить follow-up":

- следующий follow-up возвращается в очередь;
- событие логируется;
- диалог получает статус `continued`;
- в карточке видно, кто и когда продолжил.

Если пользователь нажал "Остановить цепочку":

- все будущие follow-up отменяются;
- диалог получает статус `stopped`;
- событие логируется;
- повторный запуск возможен только вручную.

## 10. Неоднозначная привязка ответов

Если один email участвует в нескольких рассылках, сервис не должен автоматически класть ответ в случайную или последнюю цепочку.

Приоритет привязки:

1. `Message-ID / In-Reply-To / References` — точная привязка.
2. Совпадение email + нормализованная тема — слабая привязка.
3. Единственная цепочка у email — слабая привязка.
4. Несколько возможных цепочек без технической ссылки — не привязывать автоматически.

Для слабой привязки интерфейс должен показывать предупреждение.

Для непривязанного ответа пользователь должен иметь возможность:

- выбрать правильную цепочку;
- оставить как отдельный диалог;
- закрыть как нерелевантное;
- добавить комментарий.

## 11. Фильтры и сортировка

Фильтры:

- все требующие решения;
- интерес;
- отказ;
- автоответ;
- не целевой;
- неуверенная привязка;
- не привязано к цепочке;
- нужно ответить вручную;
- вернуться позже;
- по кампании/импорту;
- по сегменту;
- по mailbox;
- по дате ответа.

Сортировка по умолчанию:

1. Новые живые ответы.
2. Неуверенная/непривязанная цепочка.
3. Позитивные ответы.
4. Остальные ответы.
5. Автоответы.

## 12. События и аудит

Каждое ручное действие должно писать событие:

- `reply_review_opened`;
- `reply_classification_changed`;
- `reply_thread_assigned`;
- `reply_thread_unassigned`;
- `outreach_conversation_stopped`;
- `outreach_conversation_continued`;
- `outreach_followup_delayed`;
- `manual_reply_drafted`;
- `manual_reply_sent`;
- `suppression_added_from_reply`;
- `reply_review_closed`.

В событии нужно хранить:

- кто/когда сделал действие;
- lead_id;
- campaign_id/import_id;
- message_id;
- старый статус;
- новый статус;
- причину;
- сколько писем в очереди отменено/возобновлено.

## 13. Экспорт для ИИ

Из раздела "Требуют решения" должен быть экспорт:

- выбранных диалогов;
- всех диалогов review;
- только позитивных;
- только неуверенных;
- только непривязанных.

Формат: JSONL.

В каждой строке:

- данные лида;
- сегмент;
- кампания/импорт;
- статусы;
- все отправленные письма;
- все входящие ответы;
- очищенный текст ответов;
- качество привязки;
- события и решения пользователя;
- заметки.

## 14. Критерии приемки

Раздел считается готовым, если:

- пользователь после рассылки открывает "Требуют решения" и сразу видит, какие ответы нужно обработать;
- ответы сгруппированы по диалогам, а не свалены списком писем;
- видно, к какой рассылке относится ответ;
- видно качество привязки ответа;
- неоднозначные ответы не раскладываются автоматически по чужим рассылкам;
- можно остановить, продолжить или перенести follow-up;
- можно классифицировать ответ;
- можно закрыть диалог;
- все действия пишутся в события;
- после действия диалог исчезает из review или меняет статус;
- данные можно выгрузить для AI-анализа.

## 15. Подключение ИИ для анализа ответов

ИИ нужен не для автоматической отправки писем вместо пользователя, а для помощи в разборе ответов и снижении ручной работы.

Первый этап внедрения:

- классифицировать входящий ответ;
- определять этап воронки;
- оценивать качество лида;
- оценивать, относится ли ответ к конкретной рассылке;
- объяснять, почему система считает привязку точной или сомнительной;
- предлагать следующее действие пользователю;
- готовить краткое резюме диалога.

ИИ не должен сам:

- отправлять ответ получателю;
- продолжать follow-up после живого ответа;
- удалять или закрывать диалог;
- добавлять email/домен в стоп-лист без подтверждения;
- привязывать спорный ответ к рассылке без пометки уверенности.

### 15.1. Провайдер ИИ

В качестве первого провайдера можно использовать Yandex GPT по аналогии с проектом `kusidis`.

Из `kusidis` полезно забрать не бизнес-логику, а общий подход:

- отдельный сервис-клиент для Yandex GPT;
- получение IAM-токена из OAuth-токена;
- кеширование IAM-токена;
- единая функция вызова модели;
- единый парсинг usage;
- mock-режим для тестов без внешней сети.

Секреты нельзя хардкодить в коде или коммитить в репозиторий.

Переменные окружения:

- `YANDEX_FOLDER_ID` — id каталога Yandex Cloud;
- `YANDEX_IAM_TOKEN` — короткоживущий IAM-токен, если задан напрямую;
- `YANDEX_PASSPORT_AUTH_TOKEN` — OAuth-токен, из которого сервис получает IAM-токен;
- `YANDEX_GPT_MODEL` — модель по умолчанию, например `yandexgpt-lite`;
- `YANDEX_GPT_MOCK=1` — режим тестов без реального запроса к Yandex GPT.

Если `YANDEX_IAM_TOKEN` и `YANDEX_PASSPORT_AUTH_TOKEN` не заданы, ИИ-функции должны быть выключены, а интерфейс должен показывать понятное сообщение: "ИИ-анализ не настроен".

### 15.2. Где ИИ включается в обработке входящих

Обработка входящего письма должна идти по уровням:

1. Техническая привязка по `Message-ID`, `In-Reply-To`, `References`.
2. Слабая привязка по email + нормализованной теме.
3. Слабая привязка по единственной активной цепочке у email.
4. ИИ-анализ только если технической уверенности недостаточно.
5. Если ИИ не уверен — отправить письмо в "Требуют решения".

ИИ не должен перебивать точную техническую привязку.

Если `Message-ID / In-Reply-To / References` дали точное совпадение, ИИ можно использовать только для классификации смысла ответа, но не для смены цепочки.

### 15.3. ИИ-привязка ответа к рассылке

ИИ должен получать ограниченный контекст:

- очищенный текст входящего ответа;
- тему входящего письма;
- email отправителя;
- список возможных цепочек по этому email;
- для каждой цепочки:
  - название импорта/кампании;
  - тема первого письма;
  - текст первого письма;
  - тексты follow-up;
  - последние 1-3 сообщения диалога;
  - даты отправки.

ИИ должен вернуть только JSON:

```json
{
  "best_thread_id": "uuid или null",
  "confidence": 0.82,
  "classification": "positive_reply",
  "next_action": "reply_manually",
  "reason": "Ответ про оплату совпадает с темой и текстом рассылки о задолженности",
  "needs_human_review": false
}
```

Правила:

- `confidence >= 0.85` — можно автоматически предложить привязку, но все равно сохранить пометку `ai_assisted`;
- `confidence >= 0.65` и `< 0.85` — показать как вероятную привязку и отправить в "Требуют решения";
- `confidence < 0.65` — не привязывать автоматически;
- если есть несколько близких вариантов — `best_thread_id = null`, `needs_human_review = true`;
- причина должна быть короткой и понятной по-русски.

### 15.4. Классификация ответа

ИИ должен классифицировать ответ в один из статусов:

- `positive_reply` — интерес, согласие, готовность обсудить, просьба о деталях;
- `negative_reply` — отказ, "не интересно", "не актуально";
- `neutral_reply` — нейтральный ответ без явного интереса или отказа;
- `not_target` — написал не тот человек, не тот отдел, не та компания;
- `auto_reply` — автоответ, отпуск, out of office, системное уведомление;
- `bounce` — недоставка;
- `unsubscribe` — просьба больше не писать;
- `unknown` — ИИ не уверен.

ИИ должен отдавать:

```json
{
  "classification": "positive_reply",
  "confidence": 0.91,
  "reason": "Получатель просит прислать подробности и готов обсудить"
}
```

Если уверенность ниже `0.75`, ответ должен попадать в "Требуют решения".

### 15.5. Резюме диалога и следующее действие

Для каждого диалога ИИ может подготовить:

- краткое резюме: 1-3 предложения;
- что хочет получатель;
- риск/сомнение;
- рекомендуемое действие;
- черновик ручного ответа.

Черновик ответа нельзя отправлять автоматически. Пользователь должен открыть, проверить и нажать отправку вручную.

Пример:

```json
{
  "summary": "Получатель уточняет сроки оплаты и не отказывается от диалога.",
  "recipient_intent": "нужна конкретика",
  "recommended_action": "reply_manually",
  "reply_draft": "Иван, понял. Давайте зафиксируем срок оплаты до пятницы. Подойдёт?"
}
```

### 15.6. Хранение результата ИИ

Для входящих сообщений и диалогов нужно хранить:

- `ai_classification`;
- `ai_confidence`;
- `ai_reason`;
- `ai_suggested_thread_id`;
- `ai_thread_confidence`;
- `ai_next_action`;
- `ai_summary`;
- `ai_model`;
- `ai_usage`;
- `ai_analyzed_at`;
- `ai_error`, если анализ не выполнен.

Это нужно, чтобы:

- показывать пользователю причину решения;
- не гонять один и тот же ответ через ИИ много раз;
- учитывать стоимость запросов;
- переанализировать только изменившиеся или спорные диалоги.

### 15.7. Интерфейс

В "Требуют решения" нужно показывать ИИ не как "истину", а как подсказку:

- "ИИ считает: интерес";
- "Уверенность: 91%";
- "Почему: получатель просит подробности";
- "Привязка к рассылке: вероятно, но нужна проверка";
- кнопка "Принять подсказку";
- кнопка "Исправить";
- кнопка "Переанализировать".

Если ИИ не настроен:

- раздел должен продолжать работать без ИИ;
- техническая привязка и ручная классификация должны остаться доступными;
- пользователь должен видеть: "ИИ-анализ не настроен. Можно работать вручную".

### 15.8. Ограничения и безопасность

Нельзя отправлять в ИИ лишние данные:

- пароли mailbox;
- SMTP/IMAP настройки;
- токены;
- внутренние технические логи без необходимости;
- данные других кампаний, которые не являются кандидатами для привязки.

Перед отправкой в ИИ текст нужно ограничивать по длине:

- входящий ответ — до 4000 символов;
- каждое письмо цепочки — до 3000 символов;
- максимум 5 возможных цепочек за один запрос;
- длинную историю обрезать до последних сообщений.

Если кандидатов больше 5, сначала нужно отфильтровать их эвристикой по email, теме, датам и последней активности.

### 15.9. Критерии приемки ИИ-слоя

ИИ-слой считается готовым, если:

- сервис работает без ИИ, если переменные окружения не заданы;
- Yandex GPT вызывается через отдельный сервис, а не напрямую из обработчиков;
- техническая привязка по `Message-ID` остается приоритетной;
- ИИ включается для слабых и спорных случаев;
- результат ИИ сохраняется в базе;
- пользователь видит классификацию, уверенность и причину;
- спорные ответы попадают в "Требуют решения";
- ИИ не отправляет письма без подтверждения;
- есть mock-режим для тестов;
- ошибки Yandex GPT не ломают IMAP-синхронизацию.

## 16. ИИ как помощник по воронке

ИИ должен помогать пользователю понимать не только "что написал получатель", но и "где этот контакт сейчас находится в воронке".

Главная задача:

- быстро отделять интересные ответы от шума;
- показывать, кому нужно ответить в первую очередь;
- фиксировать причину отказа или интереса;
- помогать планировать следующий шаг;
- готовить данные для последующего анализа рассылок.

### 16.1. Этапы воронки

Для каждого диалога нужно хранить `funnel_stage`.

Минимальный набор этапов:

- `new` — письмо отправлено, ответа еще нет;
- `opened_unknown` — есть технический сигнал, но ответа нет;
- `replied_neutral` — получен нейтральный ответ;
- `interested` — есть интерес;
- `details_requested` — получатель просит подробности;
- `price_requested` — получатель спрашивает цену/условия;
- `meeting_possible` — можно предложить встречу или созвон;
- `delegated` — получатель передал другому человеку;
- `follow_up_later` — попросили написать позже;
- `not_now` — сейчас не актуально, но не окончательный отказ;
- `rejected` — отказ;
- `not_target` — не тот человек или компания;
- `unsubscribed` — попросили не писать;
- `bounced` — письмо не доставлено;
- `closed` — диалог закрыт.

В интерфейсе этапы должны отображаться по-русски:

- "Нет ответа";
- "Нейтральный ответ";
- "Есть интерес";
- "Просит подробности";
- "Спрашивает цену";
- "Можно вести к встрече";
- "Передал другому";
- "Написать позже";
- "Не сейчас";
- "Отказ";
- "Не целевой";
- "Не писать";
- "Недоставка";
- "Закрыто".

### 16.2. Качество лида

ИИ должен рассчитывать `lead_temperature`:

- `hot` — горячий;
- `warm` — теплый;
- `cold` — холодный;
- `bad_fit` — не подходит;
- `unknown` — недостаточно данных.

Пример логики:

- горячий — просит цену, встречу, детали, предлагает обсудить;
- теплый — отвечает заинтересованно, но без конкретного следующего шага;
- холодный — нейтральный ответ или "может быть потом";
- не подходит — не тот контакт, не тот сегмент, явный отказ;
- неизвестно — автоответ, короткий непонятный ответ, мало контекста.

В карточке диалога нужно показывать:

- температуру лида;
- уверенность ИИ;
- причину оценки.

### 16.3. Причины отказа и интереса

Для аналитики нужно хранить `reply_reason`.

Причины интереса:

- `wants_details` — просит подробности;
- `wants_price` — спрашивает цену;
- `wants_case` — просит кейсы/примеры;
- `wants_call` — готов к созвону;
- `has_problem` — описывает проблему;
- `delegated` — передал ответственному;
- `timing_later` — просит вернуться позже.

Причины отказа:

- `not_relevant` — не актуально;
- `no_budget` — нет бюджета;
- `has_vendor` — уже есть подрядчик;
- `wrong_person` — не тот человек;
- `wrong_company` — не та компания;
- `too_expensive` — дорого;
- `bad_timing` — не вовремя;
- `negative_reaction` — негативная реакция;
- `do_not_contact` — просит больше не писать.

Если причина не определена, использовать `unknown`.

### 16.4. Next best action

ИИ должен предлагать `next_best_action`, но не выполнять его автоматически.

Возможные действия:

- `reply_manually` — ответить вручную;
- `send_details` — отправить подробности;
- `send_price` — отправить цену/условия;
- `send_cases` — отправить кейсы;
- `suggest_call` — предложить созвон;
- `ask_qualifying_question` — задать уточняющий вопрос;
- `follow_up_later` — вернуться позже;
- `stop_sequence` — остановить цепочку;
- `mark_not_target` — отметить как нецелевого;
- `add_to_suppression` — добавить в стоп-лист;
- `choose_thread` — выбрать правильную цепочку;
- `no_action` — действий не требуется.

В интерфейсе next action должен быть на русском и с объяснением:

> "Рекомендуется: ответить вручную. Причина: человек просит подробности и не отказался от предложения."

### 16.5. Черновик ручного ответа

ИИ может готовить черновик ответа, но только как подсказку.

Требования:

- черновик не отправляется автоматически;
- пользователь может отредактировать текст;
- отправка только после ручного подтверждения;
- черновик должен учитывать последний ответ, всю цепочку и стиль исходного письма;
- черновик должен быть коротким и человеческим;
- если данных мало, ИИ должен предложить уточняющий вопрос, а не выдумывать детали.

Пример:

```json
{
  "reply_draft": "Иван, да, могу прислать примеры. Правильно понимаю, вам интереснее посмотреть кейсы по B2B-аутричу или по автоматизации рассылок?",
  "draft_goal": "уточнить интерес и продолжить диалог",
  "needs_user_edit": true
}
```

### 16.6. Проверка письма перед отправкой

ИИ может проверять исходящие письма до запуска рассылки.

Проверка должна подсвечивать:

- письмо слишком шаблонное;
- нет персонализации;
- слишком длинный текст;
- непонятно, что делать получателю;
- слишком агрессивный тон;
- есть орфографические ошибки;
- есть пустые переменные;
- тема письма не совпадает с текстом;
- follow-up не связан с предыдущим письмом;
- письмо похоже на массовый спам.

Результат проверки:

```json
{
  "quality_score": 82,
  "issues": [
    {
      "level": "warning",
      "message": "Нет явного следующего шага для получателя"
    }
  ],
  "suggestions": [
    "Добавить короткий вопрос в конце письма"
  ]
}
```

Проверка не должна блокировать отправку без критичной причины. Критичные причины:

- пустой email;
- пустой текст письма;
- пустые обязательные переменные;
- не выбран mailbox;
- письмо содержит технические placeholders вроде `{{company}}`.

### 16.7. Аналитика после рассылки

После отправки ИИ должен помогать анализировать кампанию:

- какие письма дали больше ответов;
- какие сегменты отвечают лучше;
- какие темы чаще приводят к интересу;
- какие формулировки вызывают отказ;
- какие follow-up реально двигают диалог;
- где письма выглядят слишком шаблонно;
- какие причины отказов повторяются;
- какие лиды стоит обработать в первую очередь.

Минимальный отчет по кампании:

```json
{
  "campaign_summary": "Лучше всего ответили компании из сегмента ресторанов. Основной интерес — автоматизация повторных касаний.",
  "best_segments": ["Рестораны", "B2B услуги"],
  "weak_points": ["В первом письме не хватает конкретного кейса"],
  "top_objections": ["не актуально", "уже есть подрядчик"],
  "recommended_changes": [
    "Добавить короткий кейс в первое письмо",
    "Сделать follow-up короче"
  ]
}
```

### 16.8. Какие данные хранить

Для диалога нужно хранить дополнительные поля:

- `funnel_stage`;
- `lead_temperature`;
- `reply_reason`;
- `next_best_action`;
- `ai_reply_draft`;
- `ai_quality_score`;
- `ai_summary`;
- `ai_recommendations`;
- `ai_confidence`;
- `ai_reason`;
- `ai_updated_at`.

Для кампании/импорта:

- `ai_campaign_summary`;
- `ai_best_segments`;
- `ai_top_objections`;
- `ai_recommended_changes`;
- `ai_analyzed_at`.

### 16.9. Приоритет внедрения

Порядок реализации:

1. Классификация ответа.
2. Этап воронки.
3. Next best action.
4. Краткое резюме диалога.
5. ИИ-привязка спорных ответов к рассылке.
6. Черновик ручного ответа.
7. Проверка письма перед отправкой.
8. Аналитика кампании после рассылки.

Такой порядок даст пользу быстро: пользователь сначала перестает вручную разбирать входящие, потом получает подсказки по действиям, и только после этого — глубокую аналитику.
