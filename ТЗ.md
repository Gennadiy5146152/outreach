# Техническое задание: локальный сервис для личного B2B-аутрича

## 1. Назначение

Сервис нужен для личного использования и локальной работы с B2B-аутричем: хранить базу потенциальных клиентов, валидировать email, готовить и отправлять цепочки писем, читать входящие, фиксировать ответы, прогревать подключенные почтовые ящики, трекать открытия и считать метрики эффективности.

Первый этап должен быть не игрушечной таблицей, а полноценной локальной системой:

- локальный web-интерфейс;
- локальный backend;
- локальная база данных PostgreSQL;
- фоновые задачи для почты, валидации, прогрева, трекинга и аналитики.

Сервис работает для одного пользователя и не предназначен для публичного multi-tenant SaaS.

## 2. Цель первого этапа

Собрать рабочую локальную систему, которая позволяет понять, работает ли выбранный подход к аутричу:

- загрузить или вручную собрать базу лидов;
- проверить качество email-адресов;
- сегментировать лидов;
- создать кампанию и цепочку писем;
- отправлять письма через подключенные почтовые ящики с лимитами;
- читать входящие и определять ответы;
- трекать открытия писем;
- прогревать собственные подключенные почтовые ящики;
- хранить все события в PostgreSQL;
- смотреть метрики по кампаниям, почтам, сегментам и шаблонам.

## 2.1. Актуализированная продуктовая модель: персональный аутрич из Excel

Основная цель сервиса — не массовая рассылка ради рассылки, а управляемый персональный B2B-аутрич.

Целевой рабочий процесс:

1. Пользователь собирает базу и готовит индивидуальные письма в Excel.
2. Excel загружается в сервис.
3. Каждая строка Excel становится персональным черновиком письма или цепочки писем.
4. Пользователь проверяет черновики, ошибки, отправителя, лимиты и расписание.
5. Сервис отправляет письма аккуратно, через подключенные mailbox, с задержками и лимитами.
6. Сервис читает входящие через IMAP и собирает переписку в цепочки по каждому контакту.
7. Если контакт отвечает, автоматическая цепочка останавливается и требует решения пользователя.
8. Все диалоги можно выгружать для анализа с помощью ИИ.

Ключевое отличие от классической email-рассылки:

- письмо может быть полностью индивидуальным, а не собранным из одного общего шаблона;
- Excel является основным источником персонализации;
- кампания является контейнером для пакета персональных писем и цепочек;
- главный объект анализа — диалог с конкретным email/компанией, а не только отправленное письмо;
- после ответа система не должна продолжать писать автоматически без разрешения пользователя.

### 2.1.1. Excel как источник персональных писем

Сервис должен поддерживать импорт `.xlsx` и `.csv`.

Одна строка файла может описывать:

- одного лида;
- первое персональное письмо;
- follow-up шаги;
- желаемый mailbox отправки;
- дату/время старта;
- служебные заметки.

Минимальные колонки Excel:

- `email` — email получателя;
- `company` — компания;
- `contact` — имя контакта;
- `subject` — тема первого письма;
- `body` — текст первого письма;
- `segment` — сегмент;
- `mailbox` — желаемый отправляющий ящик, если нужно;
- `send_after` — не отправлять раньше этой даты/времени;
- `notes` — заметки пользователя.

Дополнительные колонки для цепочки:

- `followup_1_subject`;
- `followup_1_body`;
- `followup_1_delay_days`;
- `followup_2_subject`;
- `followup_2_body`;
- `followup_2_delay_days`;
- `followup_3_subject`;
- `followup_3_body`;
- `followup_3_delay_days`.

Допускается два режима импорта цепочки:

1. `wide format` — все follow-up шаги находятся в одной строке Excel.
2. `template sequence` — Excel содержит первое письмо и ссылку на выбранную последовательность follow-up шагов в сервисе.

После импорта пользователь должен видеть экран проверки:

- сколько строк найдено;
- сколько строк готовы к отправке;
- сколько строк с ошибками;
- какие email невалидны;
- какие строки являются дублями;
- у каких строк нет темы или текста;
- какие mailbox не найдены;
- какие письма будут отправлены первыми;
- какие follow-up шаги будут запланированы.

Импорт не должен сразу отправлять письма. Он создает черновики и план отправки.

### 2.1.2. Черновики персональных писем

После импорта сервис должен создавать черновики.

Черновик письма хранит:

- `lead_id`;
- `campaign_id`;
- `sequence_id` или `sequence_step_id`;
- `mailbox_id`;
- `to_email`;
- `subject`;
- `body_text`;
- `body_html`;
- `attachments`;
- `send_after`;
- `status`;
- `source_import_id`;
- `source_row_number`;
- `created_at`;
- `updated_at`.

Статусы черновика:

- `draft` — создан, еще не готов;
- `ready` — готов к постановке в очередь;
- `blocked` — есть ошибка, отправлять нельзя;
- `queued` — поставлен в очередь;
- `sent` — отправлен;
- `cancelled` — отменен;
- `skipped` — пропущен пользователем.

Пользователь должен иметь возможность:

- открыть черновик;
- отредактировать тему и текст;
- выбрать mailbox;
- включить/выключить tracking;
- проверить переменные и вложения;
- поставить один черновик в очередь;
- поставить пачку готовых черновиков в очередь;
- отменить черновик.

### 2.1.3. Цепочки писем

Сервис должен поддерживать последовательности писем по шагам.

Последовательность состоит из:

- первого письма;
- follow-up 1;
- follow-up 2;
- follow-up 3;
- опциональных дополнительных шагов.

Каждый шаг содержит:

- номер шага;
- задержку после предыдущего шага;
- тему;
- текст;
- вложения;
- режим подтверждения перед отправкой.

Правило автоматической отправки:

- пока от контакта нет ответа, сервис может автоматически отправлять следующие шаги цепочки;
- если пришел любой входящий ответ от контакта, автоматическая цепочка останавливается;
- будущие письма по этой цепочке переводятся из `pending/retrying` в `cancelled_reply_received` или аналогичный отмененный статус;
- дальнейшие письма этому контакту можно отправлять только после решения пользователя.

### 2.1.4. Реакция на ответы и ручное разрешение продолжения

Когда сервис получает входящее письмо, он должен:

1. Найти связанный диалог по `Message-ID`, `In-Reply-To`, `References`, email, кампании и теме.
2. Сохранить входящее письмо в историю.
3. Классифицировать ответ.
4. Остановить автоматическую цепочку для этого контакта.
5. Отменить будущие queued follow-up письма.
6. Поставить диалог в список "Требуют решения".

Статусы диалога:

- `active_sequence` — цепочка идет автоматически;
- `waiting_reply_review` — пришел ответ, нужно решение пользователя;
- `manual_reply_needed` — нужно ответить вручную;
- `paused` — остановлено пользователем;
- `completed` — цепочка завершена;
- `positive` — есть интерес;
- `negative` — отказ;
- `not_target` — нецелевой контакт;
- `bounced` — недоставка;
- `unsubscribed` — отказ от коммуникации.

Решения пользователя по входящему ответу:

- `Остановить цепочку`;
- `Продолжить цепочку`;
- `Ответить вручную`;
- `Сгенерировать ответ с ИИ`;
- `Отложить follow-up`;
- `Отметить как положительный ответ`;
- `Отметить как отказ`;
- `Отметить как автоответ`;
- `Добавить в стоп-лист`.

Автоответы обрабатываются отдельно:

- обычный автоответ не должен считаться положительным или живым ответом;
- если автоответ содержит дату возвращения, сервис должен предложить перенести следующий follow-up на дату после возвращения;
- если дата не найдена, сервис может предложить продолжить цепочку по расписанию или сдвинуть следующий шаг на несколько дней;
- решение по автоответу должно быть видно пользователю.

### 2.1.5. Диалоги и история переписки

Для каждого email/лида должен быть единый диалог.

Диалог объединяет:

- исходящие письма;
- follow-up письма;
- входящие ответы;
- автоответы;
- bounce;
- ручные заметки;
- события открытия;
- решения пользователя.

Диалог должен показывать:

- компанию;
- email;
- сегмент;
- источник импорта;
- текущий статус;
- последнее действие;
- следующий запланированный шаг, если он есть;
- полную историю сообщений в хронологическом порядке;
- классификацию ответа;
- AI summary, если он создан;
- next action.

Диалог должен быть главным объектом для анализа, потому что именно он показывает, что реально произошло с конкретным контактом.

### 2.1.6. Экспорт диалогов для анализа ИИ

Сервис должен уметь выгружать диалоги для внешнего AI-анализа.

Форматы экспорта:

- `jsonl` — основной формат для ИИ;
- `json` — полный экспорт;
- `csv` — плоская таблица для Excel;
- `xlsx` — удобный экспорт для ручного анализа.

Минимальная структура `jsonl`:

```json
{
  "lead": {
    "email": "client@example.com",
    "company": "Компания",
    "contact": "Имя",
    "segment": "Сегмент"
  },
  "campaign": {
    "name": "Июль / рестораны",
    "source_import": "file.xlsx"
  },
  "dialogue": {
    "status": "positive",
    "classification": "positive_reply",
    "messages": [
      {
        "direction": "outbound",
        "type": "outreach",
        "subject": "Тема",
        "body": "Текст",
        "sent_at": "2026-07-15T10:00:00Z"
      },
      {
        "direction": "inbound",
        "type": "reply",
        "subject": "Re: Тема",
        "body": "Ответ",
        "received_at": "2026-07-15T12:00:00Z",
        "classification": "positive_reply"
      }
    ]
  }
}
```

Экспорт должен уметь фильтровать:

- по кампании;
- по сегменту;
- по mailbox;
- по статусу диалога;
- по классификации ответа;
- по дате отправки;
- по дате ответа;
- только диалоги с ответами;
- только положительные ответы;
- только негативные ответы;
- только без ответа.

### 2.1.7. Новые основные разделы интерфейса

Целевая структура интерфейса:

- `Импорт Excel` — загрузка файлов, проверка строк, ошибки, создание черновиков;
- `Черновики` — персональные письма перед отправкой;
- `Отправка` — очередь и расписание;
- `Диалоги` — цепочки переписки по каждому контакту;
- `Требуют решения` — ответы, после которых автоматическая цепочка остановлена;
- `База` — лиды и компании;
- `Почта` — mailbox и SMTP/IMAP;
- `Прогрев` — отдельная зона прогрева;
- `Аналитика` — метрики по outreach без смешивания с прогревом;
- `Экспорт для ИИ` — выгрузка диалогов и результатов.

Текущий раздел "Рассылка" должен постепенно превратиться в более понятный раздел "Отправка" или "Аутрич".

### 2.1.8. Метрики новой модели

Основные outreach-метрики должны считаться без прогрева и без тестовых писем.

Минимальные метрики:

- импортировано строк;
- создано черновиков;
- готово к отправке;
- заблокировано из-за ошибок;
- отправлено первых писем;
- отправлено follow-up;
- уникальные открытия;
- диалоги с ответом;
- уникальные лиды с ответом;
- положительные ответы;
- негативные ответы;
- автоответы;
- недоставки;
- цепочки остановлены из-за ответа;
- требуют решения;
- среднее время до ответа;
- конверсия из отправки в ответ;
- конверсия из ответа в положительный ответ.

Важно: "Ответы" на главной должны считаться по уникальным диалогам/лидам, а не по количеству технических входящих писем, чтобы автоответы и длинные переписки не раздували метрику.

### 2.1.9. Приоритет внедрения новой модели

Рекомендуемый порядок реализации:

1. Импорт `.xlsx` с персональными письмами.
2. Экран проверки импорта и ошибок.
3. Черновики персональных писем.
4. Постановка готовых черновиков в очередь.
5. Сущность диалога/conversation.
6. Связывание входящих и исходящих в единый диалог.
7. Автостоп цепочки при ответе.
8. Экран "Требуют решения".
9. Экспорт диалогов в `jsonl/csv/xlsx`.
10. AI summary и AI-рекомендации по ответам.

## 3. Пользователь

Один пользователь: владелец сервиса.

Роли, команды, биллинг и разграничение прав в первом этапе не нужны.

## 4. Архитектура первого этапа

### 4.1. Компоненты

Минимальная архитектура:

- `frontend` — локальный web-интерфейс;
- `backend API` — локальный HTTP API;
- `postgres` — основное хранилище данных;
- `worker` — фоновые задачи;
- `mail adapter` — SMTP/IMAP-интеграция;
- `tracking endpoint` — endpoint для пикселя открытия;
- `scheduler` — планировщик отправок, проверок входящих, валидации и прогрева.

### 4.2. Рекомендуемый стек

Выбранный стек первого этапа:

- Backend: Node.js.
- Frontend: простой и быстрый web UI без полноценного SPA на первом этапе.
- DB: PostgreSQL.
- Очереди: на первом этапе задачи хранятся в PostgreSQL; отдельный Redis не обязателен.
- Локальный запуск: Docker Compose для PostgreSQL, backend и worker.
- Node.js приложение запускается в Docker Compose вместе с PostgreSQL, чтобы окружение было воспроизводимым.
- HTML-редактор: выбрать готовый редактор на этапе реализации без тяжелой SPA-архитектуры; если нет сильных ограничений, использовать простой editor на базе `contenteditable` или легкой библиотеки.

### 4.3. Важное ограничение по open tracking

Для реального трекинга открытий tracking pixel должен быть доступен получателю по публичному URL.

Локальный backend может принимать открытия только если настроен публичный адрес:

- локальный туннель, например `ngrok`, `cloudflared`, `localtunnel`;
- или небольшой публичный endpoint в будущей версии.

В настройках сервиса должен быть параметр `PUBLIC_TRACKING_URL`. Если он не задан, сервис может генерировать письма без пикселя или показывать предупреждение, что открытия реальных получателей не будут фиксироваться.

## 5. Основные сценарии

### 5.1. Работа с базой

Пользователь может:

- добавить лида вручную;
- импортировать лидов из CSV;
- редактировать карточку лида;
- менять статус лида;
- фильтровать и искать лидов;
- экспортировать базу в CSV и JSON;
- видеть и объединять дубли по email, домену или компании;
- запускать валидацию email;
- видеть историю всех касаний и событий по лиду.

### 5.2. Валидация email

Сервис должен валидировать адреса до отправки.

В первом этапе используется безопасная валидация без SMTP-probe.

Минимальные уровни валидации:

- синтаксис email;
- домен существует;
- MX-записи существуют;
- определение почтового провайдера;
- определение disposable-доменов по локальному списку;
- определение role-based адресов: `info@`, `sales@`, `support@`, `hr@` и т.д.;
- статус валидации: `valid`, `invalid`, `risky`, `unknown`.

SMTP-probe в первом этапе не используется, потому что он может давать ложные результаты, создавать лишний сетевой шум и повышать риск блокировок.

Дополнительные проверки первого этапа:

- bounce history: если ранее был bounce, адрес считается рискованным или невалидным;
- локальный список одноразовых доменов;
- локальный список role-based адресов.

Результат валидации должен сохраняться в PostgreSQL с датой проверки и причиной статуса.

### 5.3. Работа с кампаниями

Пользователь может:

- создать кампанию;
- задать ICP/сегмент кампании;
- задать лимиты отправки;
- выбрать отправляющие почтовые ящики;
- создать цепочку писем;
- настроить шаги цепочки: название, задержка в днях, тема, текст;
- добавлять вложения к письмам;
- редактировать HTML-версию письма в визуальном редакторе;
- использовать переменные в теме и тексте письма;
- включить или выключить tracking pixel;
- запустить тестовый режим кампании только на собственные mailbox;
- поставить кампанию на паузу;
- остановить кампанию.

Язык писем в первом этапе: русский.

ICP и сегменты не фиксируются в ТЗ: пользователь выбирает их в ходе работы. Сервис должен позволять свободно задавать сегменты вручную и фильтровать по ним аналитику.

### 5.4. Отправка писем

Сервис должен отправлять письма через подключенные SMTP-аккаунты.

В первом этапе нужны два режима отправки:

- автоматическая отправка по расписанию и лимитам;
- ручной режим: очередь писем формируется автоматически, но пользователь подтверждает отправку пачки.

Обязательные требования:

- отправка только по валидным или разрешенным пользователем risky-адресам;
- дневной лимит на каждый mailbox должен быть настраиваемым, но по умолчанию жесткий дневной лимит может быть отключен;
- даже при отключенном дневном лимите обязательны рабочее окно, случайные задержки, авто-пауза при ошибках и учет ограничений провайдера;
- случайная задержка между письмами должна задаваться диапазоном `minDelayMinutes` / `maxDelayMinutes`;
- рекомендуемые значения по умолчанию: случайная задержка 7-18 минут между письмами на один mailbox;
- окно отправки по времени и дням недели задается пользователем;
- остановка цепочки после ответа;
- suppression list для адресов и доменов, которым нельзя писать;
- логирование каждой попытки отправки;
- ручная пауза кампании и отдельного mailbox;
- защита от случайного массового запуска.

Интерфейс отправки должен показывать:

- прогресс отправки по кампании и текущей пачке;
- сколько писем уже отправлено;
- сколько писем осталось;
- расчетное время завершения;
- таймер до следующего письма;
- текущий mailbox отправки;
- последние ошибки SMTP/валидации/suppression;
- список писем, которые не отправились, с причиной ошибки.

Письмо должно сохраняться как событие: кому, когда, с какого mailbox, какая кампания, какой шаг, какая тема, какой Message-ID.

### 5.5. Чтение входящих

Сервис должен читать входящие через IMAP для каждого подключенного mailbox.

Обязательные функции:

- периодически проверять новые письма;
- связывать входящие с отправленными письмами по `In-Reply-To`, `References`, `Message-ID`, email-адресу и теме;
- определять reply;
- определять bounce/недоставку;
- останавливать дальнейшие шаги цепочки после ответа;
- сохранять тело письма, тему, отправителя, дату и mailbox;
- показывать переписку в карточке лида;
- обновлять статус лида на `replied`, если найден живой ответ.

На первом этапе классификация ответа может быть простой:

- `reply` — обычный ответ;
- `positive_reply` — положительный ответ или интерес;
- `negative_reply` — отказ;
- `neutral_reply` — нейтральный/уточняющий ответ;
- `bounce` — недоставка;
- `auto_reply` — автоответ;
- `unsubscribe` — отказ от коммуникации;
- `not_target` — нецелевой контакт;
- `unknown` — не распознано.

Нужен автоклассификатор входящих ответов. В первом этапе допустим простой rule-based классификатор по заголовкам и ключевым словам, с возможностью ручной корректировки класса в UI.

Полный текст входящих писем должен храниться в PostgreSQL и показываться в карточке лида.

### 5.6. Прогрев почты

Прогрев входит в первый этап.

Сервис должен прогревать только почтовые ящики, которыми пользователь владеет или на использование которых у пользователя есть разрешение.

На старте планируется 2 подключенных mailbox, которые уже есть у пользователя. Прогрев должен работать между контролируемыми mailbox.

Минимальная механика:

- подключить несколько mailbox для прогрева;
- включить/выключить прогрев на mailbox;
- задать дневной лимит прогревочных писем;
- постепенно увеличивать лимит по дням;
- отправлять прогревочные письма между подключенными mailbox;
- читать входящие прогревочные письма;
- автоматически отвечать на часть прогревочных писем;
- помечать прогревочные цепочки отдельно от коммерческих кампаний;
- не смешивать прогревочные метрики с outreach-метриками.

Метрики прогрева:

- отправлено прогревочных писем;
- получено прогревочных писем;
- получено ответов;
- ошибки SMTP/IMAP;
- динамика лимитов;
- mailbox health score.

Важно: прогрев не должен отправляться случайным внешним адресам. Только контролируемый список mailbox.

### 5.7. Трекинг открытий

Сервис должен добавлять tracking pixel в письма, если для кампании включен трекинг и задан `PUBLIC_TRACKING_URL`.

В первом этапе нужен только трекинг открытий. Click tracking не входит в первый этап.

Пользователь готов использовать публичный туннель, например `ngrok` или `cloudflared`, чтобы tracking pixel был доступен извне.

Конкретный туннель не фиксируется: первый этап должен работать с любым публичным URL, заданным в `PUBLIC_TRACKING_URL`.

Механика:

- для каждого отправленного письма создается уникальный `trackingId`;
- в HTML-версию письма добавляется пиксель вида `/t/open/:trackingId.gif`;
- backend принимает запрос пикселя;
- событие открытия сохраняется в PostgreSQL;
- сохраняются дата, IP, user-agent, mailbox, кампания, lead, message;
- первый open и повторные opens различаются.

Ограничения:

- open rate неточен, потому что почтовые клиенты могут блокировать картинки или проксировать загрузки;
- Gmail/Apple Mail могут искажать IP и время открытия;
- open tracking должен использоваться как вспомогательная метрика, не как единственный показатель успеха.

### 5.8. Аналитика и метрики

Метрики обязательны в первом этапе.

Главные KPI первого этапа:

- открытия;
- ответы.

Нужны уровни аналитики:

- общая;
- по кампании;
- по сегменту;
- по mailbox;
- по шагу цепочки;
- по шаблону письма;
- по источнику базы.

Минимальные метрики:

- всего лидов;
- валидных email;
- risky email;
- invalid email;
- отправлено писем;
- доставлено, если возможно определить;
- bounce rate;
- open rate;
- unique open rate;
- reply rate;
- positive reply rate, если пользователь вручную размечает качество ответа;
- positive reply rate по автоклассификатору;
- meeting rate;
- conversion to won;
- unsubscribe rate;
- spam complaint, если удается определить;
- среднее время до ответа;
- количество касаний до ответа;
- эффективность по mailbox;
- эффективность по сегментам.

Дашборд должен отвечать на вопрос: работает ли подход, какая кампания/сегмент/почта/шаблон работает лучше, где просадка.

## 11. Почта и безопасность отправки

### 11.1. Подключение mailbox

Пользователь должен иметь возможность подключить mailbox через SMTP/IMAP-настройки.

Первые поддерживаемые провайдеры:

- Яндекс;
- корпоративный SMTP/IMAP.

План первого теста: 2 mailbox.

Для каждого mailbox нужно проверить:

- SMTP login;
- IMAP login;
- возможность отправить тестовое письмо;
- возможность прочитать тестовое письмо;
- MX/SPF/DKIM/DMARC отправляющего домена.

### 11.2. Лимиты

Обязательные лимиты:

- дневной лимит на mailbox как настраиваемое поле, допускается значение `unlimited`;
- дневной лимит на кампанию как настраиваемое поле, допускается значение `unlimited`;
- минимальная задержка между письмами;
- случайный jitter;
- лимит новых отправок после ошибок;
- авто-пауза mailbox при серии ошибок.

Даже если выбран `unlimited`, отправка не должна идти непрерывным потоком. Система обязана соблюдать:

- рабочее окно отправки;
- случайную задержку между письмами;
- паузу между пачками;
- авто-паузу при SMTP/IMAP ошибках;
- suppression list;
- preflight-check.

Рекомендуемые настройки по умолчанию:

- `daily_send_limit`: `unlimited`;
- `minDelayMinutes`: 7;
- `maxDelayMinutes`: 18;
- `send_window`: настраивается пользователем;
- `send_days`: настраивается пользователем.

### 11.3. Защита от нежелательной отправки

Перед запуском активной кампании сервис должен показывать preflight-check:

- сколько лидов будет затронуто;
- сколько адресов valid/risky/invalid;
- какие mailbox используются;
- какие дневные лимиты;
- включен ли open tracking;
- есть ли suppression-пересечения;
- есть ли пустые переменные в шаблонах.

Критичные ошибки должны блокировать запуск:

- нет активного mailbox;
- SMTP или IMAP не проверены;
- отправляющий домен не прошел обязательную проверку MX/SPF/DKIM/DMARC;
- нет лидов для отправки;
- все лиды invalid или suppressed;
- есть пустые обязательные переменные;
- вложение отсутствует на диске или превышает допустимый размер;
- `PUBLIC_TRACKING_URL` не задан при включенном open tracking.

### 11.4. Тестовый режим

Перед реальным запуском кампании пользователь может включить тестовый режим.

В тестовом режиме:

- письма отправляются только на собственные mailbox пользователя;
- можно проверить тему, HTML, plain text, вложения и tracking pixel;
- можно проверить IMAP-синхронизацию и связывание ответов;
- тестовые события помечаются как `test` и не смешиваются с боевой аналитикой;
- после успешного теста пользователь может запустить кампанию в боевом режиме.

## 12. Трекинг и события

Каждое важное действие должно попадать в event log.

События используются для:

- аналитики;
- отладки;
- восстановления истории;
- понимания, почему лид перешел в конкретный статус.

Open tracking должен хранить raw-события, а UI должен показывать как raw opens, так и unique opens.

## 14. Нефункциональные требования

- Сервис запускается локально.
- Основное хранилище — локальный PostgreSQL.
- Данные не уходят во внешние сервисы, кроме подключенных пользователем почтовых провайдеров и публичного tracking endpoint/tunnel, если он включен.
- Секреты не хранятся в git.
- Секреты mailbox хранятся в локальном `.env`; файл `.env` не коммитится.
- В репозитории должен быть `.env.example`.
- Все пользовательские данные экранируются при отображении.
- Массовую отправку нельзя запустить случайно.
- Все фоновые задачи должны логировать ошибки.
- Сервис должен уметь корректно продолжить работу после перезапуска.
- Вложения должны храниться локально и не попадать в git.
- HTML письма должны иметь fallback plain text.
- Ошибки отправки должны быть видны пользователю без просмотра логов backend.

## 15. Комплаенс и ограничения

Сервис предназначен для деловой B2B-коммуникации.

Обязательные защитные механики:

- suppression list;
- ручная возможность исключить email или домен;
- детект отказов от коммуникации;
- остановка цепочки после ответа;
- остановка цепочки после unsubscribe;
- понятный sender identity;
- запрет на подмену отправителя;
- лимиты отправки;
- случайные задержки между письмами;
- рабочее окно отправки;
- проверка SPF/DKIM/DMARC;
- журнал событий.

Тексты писем, правовые основания коммуникации и соблюдение применимых требований пользователь контролирует самостоятельно. Сервис должен помогать не писать тем, кто отказался или кому писать рискованно.

## 16. Критерии приемки первого этапа

Первый этап считается готовым, если пользователь может:

1. Запустить сервис локально вместе с PostgreSQL.
2. Создать и проверить подключение mailbox по SMTP/IMAP.
3. Добавить лида вручную.
4. Импортировать CSV с лидами.
5. Запустить валидацию email и увидеть результат.
6. Создать кампанию и минимум два шага письма.
7. Отредактировать письмо в HTML-редакторе.
8. Добавить вложение к письму.
9. Добавить лидов в кампанию.
10. Запустить тестовый режим кампании на свои mailbox.
11. Пройти preflight-check перед боевым запуском.
12. Запустить кампанию в ручном режиме с подтверждением пачки.
13. Запустить кампанию в автоматическом режиме по рабочему окну.
14. Видеть прогресс отправки, расчетное время завершения и таймер до следующего письма.
15. Видеть ошибки по письмам, которые не отправились.
16. Отправить письма через SMTP.
17. Увидеть отправленные письма в истории лида.
18. Прочитать входящие через IMAP с полным текстом письма.
19. Автоматически связать ответ с лидом.
20. Автоматически классифицировать ответ.
21. Вручную исправить класс ответа.
22. Остановить цепочку после ответа.
23. Зафиксировать bounce и исключить проблемный адрес.
24. Включить прогрев для контролируемых mailbox.
25. Увидеть метрики прогрева.
26. Включить open tracking при заданном `PUBLIC_TRACKING_URL`.
27. Увидеть raw opens и unique opens.
28. Посмотреть аналитику по кампании, mailbox, сегменту и шагу цепочки.
29. Экспортировать данные в CSV/JSON.
30. Сделать резервную копию PostgreSQL через `pg_dump`.

## 17. Что не входит в первый этап

Не делаем в первом этапе:

- многопользовательский SaaS;
- биллинг;
- CRM-интеграции;
- продвинутую AI-персонализацию;
- автоматический парсинг сайтов;
- покупку или сбор внешних баз внутри сервиса;
- полноценную deliverability-платформу уровня коммерческих сервисов;
- SMTP-probe при валидации email;
- click tracking;
- англоязычные шаблоны как обязательный сценарий;
- публичный облачный деплой как обязательное требование.
