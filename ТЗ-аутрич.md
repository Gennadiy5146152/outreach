# Техническое задание: сервис персонального B2B-аутрича

## 1. Назначение продукта

Сервис нужен для персонального B2B-аутрича: подготовки индивидуальных писем, отправки цепочек follow-up, остановки автоматической цепочки после ответа, хранения полной истории переписки и последующего анализа диалогов с помощью ИИ.

Это не классический сервис массовой рассылки. Основная ценность продукта — помогать пользователю вести управляемые персональные касания:

1. Подготовить индивидуальные письма в Excel.
2. Загрузить Excel в сервис.
3. Проверить письма, получателей, почтовые ящики и лимиты.
4. Отправить первое письмо и follow-up цепочку.
5. Остановить автоматическую цепочку, если человек ответил.
6. Принять ручное решение по ответу.
7. Хранить всю переписку в диалогах.
8. Выгружать диалоги для AI-анализа.

## 2. Цель MVP

Собрать рабочий инструмент, который позволяет пользователю:

- загрузить Excel с персональными письмами;
- превратить строки Excel в черновики писем;
- проверить ошибки перед отправкой;
- поставить готовые письма в очередь;
- отправлять письма через подключенные SMTP mailbox;
- читать входящие через IMAP;
- собирать исходящие и входящие письма в единые диалоги;
- автоматически останавливать follow-up цепочку после ответа;
- показывать список диалогов, требующих решения;
- экспортировать диалоги для анализа ИИ.

## 3. Ключевой пользовательский сценарий

### 3.1. Подготовка

Пользователь вручную собирает базу и пишет персональные письма в Excel.

В Excel каждая строка может содержать:

- получателя;
- компанию;
- тему письма;
- тело письма;
- сегмент;
- заметки;
- mailbox отправки;
- follow-up шаги.

### 3.2. Импорт

Пользователь загружает Excel в сервис.

Сервис:

- читает строки;
- проверяет обязательные поля;
- ищет дубли;
- проверяет email;
- сопоставляет mailbox;
- создает черновики;
- показывает отчет по ошибкам.

Импорт не должен сразу отправлять письма.

### 3.3. Проверка черновиков

Пользователь видит список черновиков:

- готовые к отправке;
- с ошибками;
- пропущенные;
- уже существующие дубли.

Пользователь может открыть черновик и исправить:

- тему;
- текст;
- email;
- компанию;
- mailbox;
- дату отправки;
- follow-up шаги.

### 3.4. Отправка

Пользователь ставит готовые черновики в очередь.

Сервис отправляет письма с учетом:

- SMTP/IMAP готовности mailbox;
- лимитов mailbox;
- рабочего окна;
- задержек между письмами;
- suppression list;
- preflight-проверки;
- dry-run режима;
- open tracking, если включен.

### 3.5. Follow-up цепочка

Если получатель не ответил, сервис может отправлять follow-up шаги по расписанию.

Пример:

1. День 0 — первое письмо.
2. День 3 — follow-up 1.
3. День 7 — follow-up 2.
4. День 14 — финальное письмо.

### 3.6. Ответ получателя

Если пришел ответ от получателя:

- сервис сохраняет входящее письмо;
- связывает его с диалогом;
- классифицирует ответ;
- останавливает автоматическую цепочку;
- отменяет будущие pending/retrying follow-up письма;
- помещает диалог в раздел "Требуют решения".

После ответа сервис не должен продолжать цепочку автоматически без разрешения пользователя.

## 5. Excel-импорт

### 5.1. Минимальные колонки

Обязательные:

- `email`;
- `subject`;
- `body`.

Желательные:

- `company`;
- `contact`;
- `position`;
- `website`;
- `segment`;
- `city`;
- `mailbox`;
- `send_after`;
- `notes`;
- `source`.

### 5.2. Follow-up колонки

Поддержать wide-format:

- `followup_1_subject`;
- `followup_1_body`;
- `followup_1_delay_days`;
- `followup_2_subject`;
- `followup_2_body`;
- `followup_2_delay_days`;
- `followup_3_subject`;
- `followup_3_body`;
- `followup_3_delay_days`.

Если follow-up колонки пустые, создается только первое письмо.

### 5.3. Правила импорта

- Строки без email не импортируются.
- Строки без темы или текста получают статус `blocked`.
- Дубли по email в рамках одного импорта подсвечиваются.
- Дубли по email в базе не создают нового лида, но могут создать новый черновик, если пользователь подтвердит.
- Если mailbox из Excel не найден, черновик создается с ошибкой.
- Если `send_after` в прошлом или пустой, письмо может быть отправлено при ближайшем запуске.
- Импорт должен сохранять `source_file`, `source_row_number`, `import_id`.

## 7. Правила остановки цепочки

### 7.1. Любой живой ответ

Если пришел обычный ответ:

- conversation получает статус `waiting_reply_review`;
- enrollment/sequence для лида останавливается;
- будущие письма очереди отменяются;
- пользователь должен принять решение.

### 7.2. Positive reply

Если ответ классифицирован как `positive_reply`:

- conversation получает статус `positive`;
- будущие follow-up отменяются;
- диалог попадает в "Требуют решения";
- пользователь может ответить вручную или сгенерировать ответ.

### 7.3. Negative reply

Если ответ классифицирован как `negative_reply`:

- conversation получает статус `negative`;
- будущие follow-up отменяются;
- лид больше не должен получать автоматические письма в рамках этой кампании.

### 7.4. Not target

Если ответ классифицирован как `not_target`:

- conversation получает статус `not_target`;
- будущие follow-up отменяются;
- пользователь может пометить компанию/домен как неподходящий.

### 7.5. Unsubscribe

Если найден отказ от коммуникации:

- email или домен добавляется в suppression list;
- все будущие письма отменяются;
- conversation получает статус `unsubscribed`.

### 7.6. Auto-reply

Автоответ не должен автоматически считаться живым ответом.

Правила:

- если найдена дата возвращения, предложить перенести следующий follow-up;
- если дата не найдена, предложить продолжить по расписанию или сдвинуть follow-up;
- пользователь должен видеть автоответ отдельно;
- автоответ не должен попадать в positive reply rate.

На MVP допустим rule-based классификатор.

Классы:

- `positive_reply`;
- `neutral_reply`;
- `negative_reply`;
- `auto_reply`;
- `unsubscribe`;
- `not_target`;

Пользователь может вручную исправить классификацию.

## 11. Безопасность и защита от ошибок

Обязательные правила:

- импорт не отправляет письма сразу;
- запуск требует preflight;
- письма без email/subject/body блокируются;
- отправка идет только через проверенные mailbox;
- SMTP/IMAP должны быть проверены;
- suppression list обязателен;
- при ответе цепочка останавливается;
- future follow-up отменяются;
- прогрев не смешивается с outreach;
- test письма не смешиваются с outreach-метриками;
- секреты не хранятся в git;
- после перезапуска очередь должна продолжить работу;
- зависшие `running` задачи должны возвращаться в `retrying`.

## 12. UX и интерфейс

Все экраны сервиса должны быть на русском языке.

Интерфейс должен быть рабочим инструментом для ежедневного аутрича, а не технической админкой и не маркетинговым лендингом.

### 12.1. Язык интерфейса

Требования:

- все названия разделов, кнопок, статусов, ошибок и подсказок должны быть на русском;
- технические термины допустимы только если без них нельзя, и рядом должно быть простое объяснение;
- нельзя показывать пользователю сырой JSON как основной результат действия;
- ошибки должны объяснять, что произошло и что делать дальше;
- английские внутренние статусы (`pending`, `active`, `reply`, `warmup`) не должны выводиться без русской расшифровки;
- в интерфейсе не должно быть смешения вроде `tracking: on`, `manual: yes`, `mailbox unknown`.

### 12.2. Пошаговые сценарии

Сложные процессы должны быть разбиты на понятные шаги.

Основные пошаговые сценарии:

- импорт Excel;
- проверка импортированных строк;
- подготовка черновиков;
- запуск отправки;
- обработка ответа;
- экспорт диалогов для ИИ;
- подключение mailbox;
- проверка SMTP/IMAP;
- настройка прогрева.

Для каждого шага пользователь должен понимать:

- что сейчас происходит;
- что уже готово;
- что блокирует следующий шаг;
- что можно исправить автоматически;
- что нужно исправить вручную;
- какая следующая кнопка или действие.

### 12.3. Чистый рабочий дизайн

Дизайн должен быть спокойным, плотным и понятным.

Требования:

- без декоративных лендинговых блоков;
- без огромных кнопок, которые занимают половину экрана;
- без дублирования одинаковых действий рядом;
- без disabled-элементов без объяснения;
- без таблиц, где важные действия спрятаны;
- без перегруженных карточек внутри карточек;
- основные действия должны быть видны, второстепенные — не мешать;
- таблицы должны быть читаемыми при большом количестве строк;
- длинные email, домены, пароли, ошибки и тексты не должны вылезать за границы;
- на мобильной ширине интерфейс не должен ломаться.

### 12.4. Состояния и обратная связь

Каждое действие пользователя должно давать понятный результат.

Обязательные состояния:

- выполняется;
- успешно;
- предупреждение;
- ошибка;
- частично выполнено;
- ничего не изменилось.

Для действий отправки, проверки и импорта нужно показывать:

- сколько обработано;
- сколько успешно;
- сколько с ошибками;
- что исправлено автоматически;
- что осталось сделать;
- ссылку или кнопку на нужный раздел.

### 12.5. Главный UX-принцип

Пользователь не должен гадать, почему письмо не отправилось, почему очередь пустая, почему лид нельзя выбрать или почему цепочка остановилась.

Любое непонятное состояние должно превращаться в явное сообщение:

- причина;
- последствия;
- следующий шаг;
- действие для исправления.

## 14. Критерии приемки MVP

MVP считается готовым, если пользователь может:

1. Загрузить Excel с персональными письмами.
2. Увидеть ошибки импорта.
3. Исправить черновик письма.
4. Поставить готовые черновики в очередь.
5. Отправить первое письмо.
6. Запланировать follow-up.
7. Получить входящий ответ через IMAP.
8. Увидеть диалог с полной историей.
9. Убедиться, что future follow-up отменен после ответа.
10. Принять ручное решение по ответу.
11. Выгрузить диалоги в `jsonl`.
12. Посмотреть метрики без прогрева и тестовых писем.

## 15. Дополнения по анализу существующих outreach/sales engagement систем

После анализа подходов в sales engagement системах в ТЗ нужно добавить несколько важных продуктовых механик.

### 15.1. Enrollment-правила

Enrollment — это включение конкретного контакта в конкретную цепочку.

Правила:

- один контакт не должен одновременно находиться в двух активных цепочках одного типа;
- повторное включение контакта в новую цепочку требует ручного подтверждения;
- если email ранее bounced, контакт нельзя включить без ручного override;
- если email или домен в suppression list, включение запрещено;
- если в персональном письме есть пустые обязательные поля, enrollment блокируется;
- bulk enrollment должен иметь лимит на количество контактов за один запуск;
- при bulk enrollment пользователь должен иметь возможность открыть каждого контакта и поправить письмо до отправки.

### 15.2. Индивидуальная проверка перед bulk-запуском

При массовой постановке черновиков в очередь нужен экран проверки:

- список всех писем;
- статус готовности;
- тема;
- первые строки письма;
- получатель;
- отправляющий mailbox;
- время отправки;
- предупреждения;
- ошибки;
- чекбокс "отправлять" у каждого письма.

Нельзя прятать проблемные письма только в общем JSON. Пользователь должен видеть, что именно будет отправлено.

### 15.3. Ручные задачи внутри цепочки

Цепочка должна поддерживать не только email-шаги, но и ручные задачи.

Типы шагов:

- `automatic_email` — автоматическое письмо;
- `manual_email` — задача написать письмо вручную;
- `call_task` — задача позвонить;
- `linkedin_task` — задача написать/добавить в LinkedIn;
- `custom_task` — произвольная задача.

Для MVP можно реализовать только email, но модель данных должна быть готова к ручным задачам.

Правило:

- если шаг требует ручного действия, цепочка может ставиться на паузу до выполнения задачи;
- пользователь может выбрать "продолжить цепочку без выполнения задачи";
- ручная задача должна попадать в отдельный список дел.

### 15.4. Автоматические причины выхода из цепочки

Цепочка должна иметь понятные причины остановки.

Причины:

- `reply_received` — получен живой ответ;
- `positive_reply` — положительный ответ;
- `negative_reply` — отказ;
- `meeting_booked` — назначена встреча;
- `auto_reply` — автоответ, требуется решение;
- `bounce` — недоставка;
- `unsubscribe` — отказ от коммуникации;
- `manual_stop` — остановлено пользователем;
- `suppression_match` — email/домен попал в стоп-лист;
- `sequence_completed` — все шаги завершены.

Для каждой остановки нужно хранить:

- дату;
- причину;
- message_id, если причина связана с письмом;
- пользователя/систему, кто остановил;
- комментарий.

### 15.5. Company-level stop

Нужна настройка: останавливать ли цепочку для всех контактов компании, если один контакт из этой компании ответил.

Варианты:

- `contact_only` — остановить только конкретный email;
- `same_domain` — остановить всех с тем же доменом;
- `same_company` — остановить всех лидов компании.

По умолчанию для личного аутрича лучше `contact_only`, но пользователь должен видеть настройку.

### 15.6. Step performance

Аналитика должна показывать эффективность каждого шага цепочки.

Метрики по шагу:

- отправлено;
- открыто;
- ответили;
- positive replies;
- negative replies;
- auto replies;
- bounce;
- stopped after this step;
- среднее время до ответа после шага.

Это нужно, чтобы понимать, какое письмо в цепочке реально работает.

### 15.7. Reply review queue как главный рабочий экран

Раздел "Требуют решения" должен быть приоритетнее обычной очереди.

В нем нужны быстрые действия:

- классифицировать ответ;
- остановить цепочку;
- продолжить цепочку;
- отложить follow-up;
- создать ручную задачу;
- открыть AI draft ответа;
- отправить ответ;
- добавить в suppression.

Для каждого ответа нужно показывать:

- всю предыдущую цепочку;
- какой шаг вызвал ответ;
- сколько касаний было до ответа;
- сколько времени прошло с первого письма;
- краткое AI summary;
- рекомендуемое действие.

### 15.8. Deliverability guardrails

Сервис должен проверять не только SMTP/IMAP, но и базовую готовность отправки.

Проверки:

- SPF;
- DKIM;
- DMARC;
- MX;
- PTR/reverse DNS, если применимо;
- TLS;
- корректный `Message-ID`;
- отсутствие misleading `From`;
- отсутствие искусственных `Re:`/`Fwd:` в первом письме;
- наличие plain text версии;
- размер письма и вложений;
- отсутствие слишком больших вложений;
- наличие suppression/unsubscribe механики для коммерческих писем.

Для больших объемов и маркетинговых/подписных писем нужно поддержать:

- `List-Unsubscribe`;
- `List-Unsubscribe-Post: List-Unsubscribe=One-Click`;
- endpoint отписки;
- автоматическое добавление email/domain в suppression после unsubscribe.

### 15.9. Adaptive throttling

Отправка должна адаптироваться к ответам SMTP и качеству mailbox.

Если появляются:

- SMTP 4xx;
- deferral;
- rate limit;
- temporary failure;
- рост bounce;
- серия mailbox_error;

сервис должен:

- снизить скорость отправки;
- увеличить задержки;
- поставить mailbox на паузу;
- показать причину пользователю;
- предложить повторить позже.

### 15.10. Ramp-up и дневные лимиты

Даже если пользователь отключил жесткий дневной лимит, сервис не должен отправлять резко и непрерывно.

Нужно поддержать:

- дневной лимит на mailbox;
- дневной лимит на кампанию;
- ramp-up schedule;
- рабочие дни;
- рабочие часы;
- случайный jitter;
- максимальное число новых диалогов в день;
- отдельный лимит follow-up писем.

Пример ramp-up:

- день 1: 5 новых контактов;
- день 2: 8;
- день 3: 12;
- день 4: 15;
- дальше по настройке пользователя.

### 15.11. Missing personalization guard

Если письмо содержит переменную или placeholder без значения, оно должно блокироваться или требовать ручного подтверждения.

Проверять:

- пустое имя;
- пустую компанию;
- пустую тему;
- пустой body;
- неразрешенные `{{variable}}`;
- одинаковые тексты у большого числа получателей;
- слишком короткий текст;
- подозрительные шаблонные маркеры вроде `{{first_name}}`, если они не обработаны.

### 15.12. Threading mode

Для follow-up нужно выбирать режим:

- `reply_to_previous` — follow-up идет ответом в предыдущую ветку;
- `new_thread` — follow-up начинается новой темой.

По умолчанию для outreach лучше `reply_to_previous`, чтобы получатель видел контекст.

Для каждого сообщения нужно хранить:

- `message_id_header`;
- `in_reply_to`;
- `references_header`;
- `threading_mode`;
- `parent_message_id`.

### 15.13. AI next-best-action

AI не должен сразу отправлять письма сам.

На первом этапе AI может:

- суммировать диалог;
- выделять интерес;
- выделять возражение;
- предлагать next action;
- готовить черновик ответа;
- предлагать перенос follow-up;
- объяснять, почему диалог помечен как positive/negative/not_target.

Отправка AI-ответа всегда требует подтверждения пользователя.

### 15.14. Audit trail

Все важные решения должны попадать в историю.

Логировать:

- импорт файла;
- создание черновика;
- правку черновика;
- постановку в очередь;
- отправку;
- отмену;
- входящий ответ;
- классификацию;
- ручную переклассификацию;
- остановку цепочки;
- возобновление цепочки;
- добавление в suppression;
- AI summary;
- AI draft.

Это нужно для анализа и восстановления контекста.

### 15.15. Что добавить в MVP после ресерча

В ближайший MVP стоит добавить:

1. Экран review перед bulk-запуском.
2. Ограничение "один контакт — одна активная цепочка".
3. Missing personalization guard.
4. Reply review queue.
5. Company/domain-level stop setting.
6. Step performance.
7. Threading mode для follow-up.
8. Adaptive throttling по SMTP ошибкам.
9. List-Unsubscribe поля для коммерческих писем.
10. Audit trail решений пользователя.
